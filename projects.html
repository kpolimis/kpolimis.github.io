Projects

These are projects where I either shipped something to production or produced peer-reviewed research. The goal is enough specificity that you can evaluate the engineering and the methodology, not just the headline number.

Healthcare Multi-Agent RAG System

Clinical documentation is one of the highest-friction tasks in healthcare operations. Providers spend hours per shift on structured note-taking that follows predictable templates but requires judgment about what to include and how to categorize it. The question was whether a multi-agent LLM system could handle the templated portions reliably enough to be trusted in a live clinical environment.

Stack: Python, FastAPI, LangChain orchestration, Pinecone vector store, HIPAA-compliant Azure deployment. Agent architecture: a router agent that classifies documentation type, specialist agents per template (intake, progress notes, discharge summaries), and a validation agent that checks output against required fields before committing. No patient data touches external model APIs.

Constraints: HIPAA compliance ruled out any cloud-hosted model with retention policies, which eliminated OpenAI and Anthropic APIs in their standard configurations. All inference runs on Azure-hosted models within a compliant boundary. Latency had to stay under 8 seconds per document or providers would context-switch. The validation agent exists because hallucinated ICD codes in clinical notes are not a recoverable error.

Results: 40% reduction in provider administrative load in pilot. The measurable outcome was time-on-task for documentation, not model accuracy in isolation, which is the right metric for whether a system is actually useful.

Lessons learned: The router agent was the hardest component to get right, not the specialist agents. Misclassifying a progress note as a discharge summary produces plausible-looking but wrong output that passes automated checks. The solution was few-shot examples for the router drawn from real edge cases, not more complex routing logic.

NBA MLOps Pipeline

Sports prediction degrades faster than most ML domains because the data-generating process changes continuously: roster moves, injuries, load management, coaching changes. A model trained on October data is already stale by December. The problem is building infrastructure that updates fast enough to track those changes without introducing the failure modes that come with frequent automated retraining.

Stack: AWS Lambda for daily ingestion (NBA Stats API, Odds API), S3 raw storage, feature computation Lambda writing to DynamoDB feature store, SageMaker Pipelines for automated retraining, FastAPI serving layer. LightGBM ensemble with isotonic calibration. Python throughout.

Constraints: Training-serving skew is the dominant failure mode in systems like this. If the feature computation logic differs between the training pipeline and the inference pipeline by even a small amount (different lookback window, different handling of back-to-back games), predictions degrade silently. The fix is a shared feature store: one computation path, read by both pipelines. The NBA Stats API has availability issues, so the ingestion Lambda logs and continues rather than failing hard, and the feature store handles missing days explicitly.

Results: Automated retraining triggers when trailing-30 Brier score exceeds threshold. Pre-game win probabilities served daily for all NBA games. System runs without intervention on stable data days.

Lessons learned: The automated retraining trigger is too coarse. A single Brier threshold doesn’t account for variance in game outcomes: 0.22 on a week of close games means something different than 0.22 on a week of blowouts. The next version needs a monitoring metric that conditions on game competitiveness.

Financial Transaction Classifier

Transaction descriptions are designed for accounting ledgers, not natural language models: SQ *STREET COFFEE HOU, CHECKCARD 0413 WHOLEFDS MKT, APL* APPLE ITUNES. The vocabulary problem is that pre-trained models have rich representations of “Whole Foods Market” but none for “WHOLEFDS MKT.” The class imbalance problem is that Legal Services transactions are 0.3% of volume but the highest-cost misclassifications.

Stack: BERT-base-uncased fine-tuned on labeled transaction data. Custom preprocessing layer to strip processor prefixes and normalize common abbreviations before tokenization. Custom weighted cross-entropy loss with label smoothing (0.1) to handle class imbalance and genuine category ambiguity. ONNX export for inference. FastAPI microservice with model loaded at startup via lifespan context manager.

Constraints: The preprocessing step is more important than it looks. “SQ *STREET COFFEE HOU” → “STREET COFFEE HOUSE” recovers meaningful text the tokenizer can handle before it ever sees the model. Without it, you’re asking BERT to generalize from truncated strings with no training signal. ONNX reduced inference latency ~60% (45ms → 18ms on CPU), which matters when the microservice is in the critical path of a user-facing application.

Results: Macro F1 of 0.85 across all categories on held-out test set, including rare categories. Macro F1 rather than accuracy is the right metric here: a model that ignores Legal Services and scores 99% accuracy is useless for this application.

Lessons learned: Transaction amount should be a feature and isn’t yet. A $3.50 transaction has high prior probability of being coffee. A $3,500 transaction classified as coffee should incur a confidence penalty. The amount field is in the API contract for future use.

March Madness Prediction System

NCAA tournament prediction is structurally different from regular-season prediction. The bracket format means you’re predicting games between teams that may not have played each other all season, using whatever signal you can extract from conference schedules and efficiency metrics. The Kaggle competition is scored on Brier score, which rewards calibration, not just picking winners.

Stack: LightGBM ensemble (LightGBM + XGBoost + Random Forest, equal weight). Feature set: KenPom efficiency metrics, seed differential, conference strength, home/away/neutral splits. Expanding-window cross-validation for model selection. Isotonic regression calibration on validation set outputs.

Constraints: k-fold cross-validation is wrong for this problem. It leaks future data: a model trained on 2018 data validated on a fold that includes 2016 games has seen outcomes it shouldn’t. The correct approach is expanding-window CV: for each validation year, train only on prior years. Estimated performance difference between k-fold and expanding-window on tournament data is 15–20% overestimation with k-fold.

Results: 2026 submission completed. Florida picked to win; they exited in the Round of 32. The model performed reasonably on seeds 1–4 and poorly on mid-major upsets, which is where most bracket prediction models fail.

Lessons learned: Calibration trained on historical tournament data fails when the current tournament has an unusual upset rate. 2026 had more upsets than the historical base rate predicted. The next version will use a rolling calibration window and shrink toward a seed-based prior rather than trusting isotonic regression on the full historical distribution.

Paratransit Routing Optimization

Paratransit serves riders with disabilities who cannot use fixed-route transit. The routing problem is harder than standard vehicle routing: vehicles have wheelchair capacity constraints, riders have medical time windows (dialysis appointments cannot be missed), and the population cannot easily rebook if the system fails. The DSSG project built optimization infrastructure for a mid-sized US transit agency handling thousands of trips per day.

Stack: Python optimization pipeline using OR-Tools. Data ingestion from agency scheduling system. Constraint formulation: vehicle capacity (standard + wheelchair), pickup/dropoff time windows, driver shift constraints, depot assignment. Output: optimized manifests fed back to dispatch.

Constraints: The hardest constraint was soft time windows. Riders request pickups in a window but have hard delivery deadlines (you can be late picking someone up for dialysis, not late dropping them off). Standard routing formulations treat windows as symmetric; this one required asymmetric penalties that reflect the actual cost structure.

Results: Routing efficiency improvements for the pilot agency. Code and methodology documented as open source. The project scope was proof-of-concept at partner scale, not production deployment.

Lessons learned: The gap between an optimized routing solution and a deployed one is mostly change management, not engineering. Dispatchers had years of institutional knowledge about which riders needed extra time, which stops were unreliable, which drivers knew which neighborhoods. The model had none of that. A production version needs a feedback loop for dispatcher corrections.

forest-confidence-interval

Random forests produce point estimates without uncertainty quantification. For research applications where the question is not just “what does the model predict” but “how confident is it,” that gap matters. This package implements the jackknife and infinitesimal jackknife variance estimators for scikit-learn random forest objects.

Stack: Python, scikit-learn integration. Implements Wager, Hastie, and Efron (2014) variance estimation. Works on regression and classification forests. Published to PyPI.

Constraints: The variance estimator requires access to the in-bag samples for each tree, which scikit-learn doesn’t expose by default. The implementation infers the in-bag matrix from the forest when bootstrapping is enabled, or accepts a user-provided matrix.

Results: JOSS paper, scikit-learn-contrib inclusion, active use in research applications that need uncertainty bounds on ensemble predictions.

Lessons learned: API design for a research tool requires different tradeoffs than API design for a production tool. The primary users are researchers who need to understand what the estimator is doing, not engineers who need it to be fast. Documentation of the statistical assumptions matters more than performance optimization.

Computational Audit of Conflict Journalism (ongoing)

Herman and Chomsky’s Manufacturing Consent (1988) made a testable prediction: US media covers victims of US-backed states differently than victims of US-opposed states. The prediction has never been tested at scale with a rigorous identification strategy. This project is that test, preregistered at OSF in April 2026 before cross-conflict data collection.

Stack: GDELT DOC API ingestion (async, 8-outlet parallel), 9-strategy body recovery chain (direct → GDELT Full-Text → Media Cloud → Event Registry → Wayback → Common Crawl), spaCy NER and attribution pipeline, IHL accuracy lexicon grounded in Geneva Conventions and ICRC Commentary, Hyland (1996) epistemic hedge taxonomy, NRC Emotion Lexicon. fixest panel regression in R with outlet and event fixed effects. BH-FDR multiple comparisons correction.

Constraints: The primary identification challenge is that framing differences across outlets could reflect genuine editorial style rather than geopolitical alignment. The design controls for this with outlet fixed effects: what’s estimated is the within-outlet change in framing across conflicts, not the between-outlet difference. The IHL accuracy module uses external legal standards (Geneva Conventions, ICJ advisory opinions) rather than researcher judgment, which makes the primary claims legally grounded rather than theoretically contested.

Results: Phase 2 Ukraine corpus analysis (3,470 articles, 6 outlets, Feb 2022–Feb 2023) complete. Cross-conflict corpus (Gaza, West Bank, Iran) re-ingestion in progress. Regression not yet estimated on full cross-conflict data.

Lessons learned: The hardest methodological problem is distinguishing IHL terminology frequency from IHL terminology accuracy. Counting how often outlets use “prisoner of war” is not the same as checking whether they use it for situations that actually meet the GCIII Article 4 definition. The module needs a three-way mapping: situation type → legally correct term → outlet’s term. Frequency alone is not the test.
